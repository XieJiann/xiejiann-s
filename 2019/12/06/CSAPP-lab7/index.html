    实验简介

    PART A

    本实验的A部分是完成一个cache模拟程序csim.c，来模拟cache的各种操作，并输出hit miss eviction

    输入：数据操作

    在trace文件夹下又很多文件，代表着各种对数据的操作，其格式为：操作符 地址,大小。其中大小就表示该需要访问的字节数，操作符意义如下：

    操作符 意义
    M Modify，数据修改：L+S
    S Save，数据存储
    L Load，数据载入
    I 指令，不计入内存访问

    cache设置

    cache有三个参数：

    • -s : Number of set index bits
    • -E : Associativity
    • -b : Number of block bits

    其结构如下：

    调度过程

    组选择

    根据set index选择对应的Set

    行匹配

    遍历所有的行，如果其valid位设置，便比较Tag位。

    如果Tag为匹配，根据块偏移选择起始字节

    不命中

    如果不命中，那么如果有空的行，便读入。

    如果没有，便选择一行置换。

    置换策略是 LRU(Least Recently Used)

    附

    csim-ref中认为只要地址中tag和index 找到相符，那么就一定能读出对应长度的数据。

    但是有没有可能在cache中只含有部分数据，比如要读8字节的数据，而block里面只有4字节的数据。

    不过这也给写程序带来便利，就是我们不需要再考虑这种情况 ：)

    PART B

    part 的实验目标很简单，就是尽量利用cache，使得矩阵转置的miss尽可能的少

    具体要求如下：

    • 32 * 32: 8 points if m < 300, 0 points if m > 600
    • 64 * 64: 8 points if m < 1300, 0 points if m > 2000
    • 61 * 67: 10 points if m < 2000, 0 points if m > 3000

    附：测试程序以来valgrand，一个内存分析的软件。

    32*32

    对于一个32*1*32(8int)的cache而言，它可以装下32 * 8的矩阵。所以我们可以将其32 * 32矩阵转置分解为多个16 * 8的矩阵转置（16*8的矩阵转置相当于两个8*8的矩阵转置，所以这里我们考虑8*8）

    理论上，我们认为这种算法8次会有一次miss。而转置算法共有32*32次读和32*32次写。故其miss的次数应该为2*32*32*0.125 = 256

    同样使用上述８*８的分块，miss次数却高达4000多次。经过查询还是由于抢占冲突问题。

    由set_index可以看出，对于32*32矩阵，8*8的分块每一行都不冲突

    而对于64*64的矩阵，后4行会和前4行冲突，这就意味着，每次执行写入B的时候都会出现miss。

    即次数为 64*64+64*64/8 = 4608 左右

    解决方法

    首先，考虑使用4*4的分块。此时，读A操作miss的概率是1/8（是按照行来读，cache保留了A的4*8个数据，这样读下一个4块时还是hit），写B操作miss的概率是1/4。次数应为1536，实测1699。

    所以，为了拿到满分，需要读写的操作都miss的概率变成1/8。
