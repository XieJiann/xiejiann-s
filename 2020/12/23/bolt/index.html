Bolt db 源码阅读
发表于2020-12-23|更新于2020-12-27
|阅读量:

bolt是一个用go语言写的嵌入式 kv 数据库，其代码设计非常简洁。阅读其源码，可以学习到一个基本的kv 数据库的架构和设计。

bolt 总体设计

bolt的架构图如下：

在磁盘上，数据是以page为单位存储管理的。然后数据会被读入到内存中，内存中的数据结构名字为node。在内存中node被组织成B+树的形式，node也被分为两类，branch node和leaf node。

在数据库中，一个B+树使用bucket结构表示，其中存储者B+树的root node的指针。bucket起到的作用类似于sql数据库中的table形式，每个bucket表示为一个namespace，不同的bucket可以存储两个key一样值。

由于事务的要求，所以数据库肯定不能直接将原来的数据覆盖。所以当写入的时候，我们需要重新计算node的大小，分配对应大小的page，然后再写入，当然这是bucket层面的东西。

在node的这层抽象中，我们需要关注的是，如何将node的东西写入page中，其中关键的是对B+树的再平衡。其中主要有两个操作：

  1. spill：当一个node插入的数据太多，就需要分裂成多个page。该操作是自底向上的，因为底层的分裂可能会造成上层的溢出

  2. rebalance：当一个node删除的数据太多，就需要合并它的兄弟节点。rebalance也是自底向上，因为底层的删除，也可能会造成父节点unbalanced

bolt 中的事务

事务的结构体：

读事务流程：

  • 拷贝meta page到内存中，初始化root bucket
  • 将当前tx追加到当前数据库的事务列表db.txs中
  • bucket.get()…
  • commit，移除txs列表中对应项

写事务流程：

  • 拷贝meta page到内存中，初始化root bucket
  • 将当前tx追加到当前数据库的事务列表db.txs中
  • bucket.put…
  • commit / roll back，移除txs列表中对应项

原子性

每次事务初始化的时候都需要重新读入meta data，并初始化bucket。即从磁盘中重新初始化数据库。

当事务在修改某个内容的时候，会先拷贝进内存中修改。当需要写入的时候，会重新分配新的页写入，防止覆盖之前的数据。

当事务commit时，并按照如下顺序将数据写入磁盘：

  • node
  • freelist
  • meta data

meta data中保存了root node，所以只有meta data写入磁盘了，修改才会生效，否则之前的修改全部无效。

隔离性

bolt db支持多个读事务，和一个写事务并发，不支持多个写事务并发。

读事务

在读事务开始时，会给mmap 加上读锁

写事务当发现free page不够时，会重mmap，扩大文件重新mmap。读事务加读锁的就是为了防止该事件发生

写事务

写事务在初始化时，会加上写锁

所以只支持一个写事务

多版本

每个事务都有id:txid，保存了最大提交写事务的id，txid实际上表现的就是当前数据库的版本

  • 读事务初始化：txid == db.meta.txid
  • 写事务初始化：txid == db.meta.txid + 1

当写事务提交时，就需要更新meta数据里的txid。

在 bolt 中，txid主要用于写事务free page时。当某个写事务尝试free page时，不会立刻加入freepage list中去，而是会先pending它：

因为该free page可能正被其它的读事务使用。

那么什么时候，释放pending的page呢，主要是两个时机：

  • 该事务roll back的时候
  • 下一个写事务初始化的时候，找到db 最小的txid，释放所有小于其的事务的pending page
